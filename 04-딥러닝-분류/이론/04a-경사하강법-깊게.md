# 4.4a (보강) 경사하강법과 옵티마이저 깊게

## "Adam 한 줄로 끝났는데 더 알 게 있어요?"

지금까지 우리는 `torch.optim.Adam(model.parameters(), lr=0.001)` 한 줄로 옵티마이저를 썼어요. 솔직히 그것만 알아도 90%는 동작해요.

근데 가끔 학습이 잘 안 될 때가 있어요. 그럴 때 옵티마이저를 좀 더 깊이 알면 디버깅이 빨라져요. 이 글에서는:

1. **경사하강법(Gradient Descent)이 진짜 무엇을 하는지**
2. **Batch / Stochastic / Mini-batch의 차이**
3. **Momentum, RMSProp, Adam이 왜 좋은지**
4. **학습률 스케줄링**

---

## 1. 경사하강법의 진짜 모습

### 비유: 안개 낀 산에서 내려오기

여러분이 산에 등산을 갔는데, 갑자기 안개가 짙게 끼었어요. 길도 안 보이고 GPS도 안 됩니다. 어떻게 내려오시겠어요?

**한 발자국 디뎌 보고, 발이 더 낮은 쪽으로 가는 것**을 반복하시겠죠. 그리고 그게 끝나는 곳이 바로 골짜기 (가장 낮은 곳).

이게 **경사하강법(Gradient Descent)** 의 직관입니다.

### 수학적으로

```
업데이트:  w_new = w_old - learning_rate × gradient
                                         ↑
                                    "어느 쪽으로 갈지"
```

- `gradient`: 손실이 증가하는 방향 (=올라가는 방향)
- `-gradient`: 그 반대 (=내려가는 방향)
- `learning_rate`: 한 걸음의 크기

### 시각화

```
손실  │
     │  \                              /
     │   \                            /
     │    \   ↓ (gradient)           /
     │     \  ←  여기서                /
     │      \  -gradient 방향으로   /
     │       \   한 걸음          /
     │        \                 /
     │         \____________/
     │              ↑
     │          최저점
     └─────────────────────→ weight
```

매 step마다 **현재 위치의 기울기를 계산하고, 반대 방향으로 한 걸음**. 이걸 수렴할 때까지.

## 2. 학습률 (Learning Rate) — 한 걸음의 크기

가장 중요한 하이퍼파라미터예요. 너무 작거나 크면 큰일 나요.

### lr이 너무 작으면

```
손실  │
     │  \                          /
     │   \    · · · · · · · ·     /     ← 너무 작게 움직여서
     │    \                       /        평생 못 도달
     │     \_____________________/
     └─────────────────────────→
```

학습이 느려요. 학습은 되는데 1000 epoch 걸려요.

### lr이 적당하면

```
손실  │
     │  \                  /
     │   \   ↓             /
     │    \  ↓             /
     │     \ ↓ ↓           /
     │      \  ↓ ↓ ↓       /
     │       \      ↓ ↓ ___/
     └─────────────────────→
```

빠르게 잘 수렴해요. 보통 10~100 epoch.

### lr이 너무 크면

```
손실  │
     │  \                /         \
     │   \              /           \
     │    ↘            /             \  ↗
     │     \  ↗       /               \
     │      \         /                 \   ← 발산!
     │       \_______/
     └─────────────────────────→
```

골짜기를 못 들어가고 양쪽을 왔다 갔다, 결국 폭발 (loss가 NaN).

### 실용 가이드

```python
# Adam의 기본값
lr = 0.001    # 거의 항상 좋음

# 학습이 느림 → 키움
lr = 0.01

# Loss가 NaN/Inf → 줄임
lr = 0.0001

# SGD라면
lr = 0.01 ~ 0.1   # SGD는 더 큰 lr 필요
```

## 3. Batch / Stochastic / Mini-batch GD

### Batch Gradient Descent (전체 배치)

매 업데이트마다 **모든 학습 데이터**를 보고 그래디언트 계산.

```python
for epoch in range(n_epochs):
    pred = model(X_train)            # 전체 데이터!
    loss = loss_fn(pred, y_train)
    loss.backward()
    optimizer.step()
```

**장점:** 안정적인 그래디언트
**단점:** 느림, 메모리 많이 씀, 큰 데이터 불가
**현대 사용:** 거의 안 씀 (작은 데이터에만)

### Stochastic GD (한 샘플씩)

매 업데이트마다 **샘플 하나만** 보고 그래디언트 계산.

```python
for epoch in range(n_epochs):
    for x, y in zip(X_train, y_train):
        pred = model(x)              # 샘플 1개!
        loss = loss_fn(pred, y)
        loss.backward()
        optimizer.step()
```

**장점:** 빠름, 노이즈가 local minimum 탈출에 도움
**단점:** 그래디언트가 시끄러움 (학습 곡선 흔들림)
**현대 사용:** 거의 안 씀 (너무 노이지)

### Mini-batch GD (현대의 표준)

매 업데이트마다 **작은 배치**(32, 64, 128 등) 단위로.

```python
for epoch in range(n_epochs):
    for x_batch, y_batch in DataLoader(...):     # 64개씩!
        pred = model(x_batch)
        loss = loss_fn(pred, y_batch)
        loss.backward()
        optimizer.step()
```

**장점:** 두 방법의 균형. GPU 활용 최대.
**단점:** batch_size 튜닝 필요
**현대 사용:** **거의 100%**

### 비교표

| | Batch GD | SGD | Mini-batch GD |
|---|---|---|---|
| 한 번 업데이트에 보는 샘플 | 전체 | 1개 | 32~256 |
| 그래디언트 정확도 | 매우 높음 | 매우 낮음 | 보통 |
| 학습 속도 | 느림 | 빠르지만 시끄러움 | **빠르고 안정** |
| GPU 활용 | 낮음 | 매우 낮음 | **최적** |
| 메모리 | 큼 | 작음 | 보통 |
| 현대 사용 | X | X | **표준** |

## 4. 옵티마이저 진화

기본 SGD에 여러 개선이 더해지면서 옵티마이저가 발전했어요.

### 4-1. SGD (기본)

```
w_new = w_old - lr × gradient
```

가장 단순. 매 step마다 그래디언트 방향으로만.

### 4-2. SGD + Momentum

이전 업데이트의 **관성**을 더해요.

```
v = β × v + gradient        # 관성 (β=0.9 보통)
w_new = w_old - lr × v
```

**비유:** 산을 내려가는 공.
- 일반 SGD: 매 발자국마다 처음부터 시작
- Momentum: 점점 가속 → 작은 언덕에서 안 멈추고 굴러감

```python
optimizer = torch.optim.SGD(
    model.parameters(),
    lr=0.01,
    momentum=0.9,    # ← 추가
)
```

### 4-3. RMSProp

각 가중치마다 **개별 학습률**을 계산. 자주 변하는 가중치는 작게, 안 변하는 가중치는 크게.

```python
optimizer = torch.optim.RMSprop(model.parameters(), lr=0.001)
```

### 4-4. Adam = Momentum + RMSProp

가장 인기 있는 옵티마이저. **둘의 장점을 합침.**

```python
optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
```

내부적으로:
- Momentum 같은 1차 모멘트
- RMSProp 같은 2차 모멘트

→ **거의 모든 상황에서 잘 동작.**

### 4-5. AdamW

Adam의 weight decay 처리를 개선한 버전. **현대 표준 (Transformer 등).**

```python
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=0.001,
    weight_decay=0.01,
)
```

### 옵티마이저 비교표

| 옵티마이저 | 계산 비용 | 메모리 | 일반화 | 튜닝 어려움 | 현대 사용 |
|----------|----------|-------|--------|----------|----------|
| SGD | 가장 작음 | 작음 | **가장 좋음** (잘 튜닝하면) | 어려움 | 큰 모델/데이터 |
| SGD+Momentum | 작음 | 보통 | 좋음 | 보통 | CV 일부 |
| RMSProp | 보통 | 보통 | 보통 | 보통 | RNN |
| **Adam** | 보통 | 큼 | 보통 | **쉬움** | **거의 모든 곳** |
| **AdamW** | 보통 | 큼 | 좋음 | **쉬움** | **Transformer 표준** |

## 5. 학습률 스케줄링

학습률을 **시간에 따라 조정**하면 보통 더 좋은 결과.

### 왜?

```
학습 초기:  큰 걸음으로 빠르게 골짜기 근처로
학습 후기:  작은 걸음으로 정밀하게 바닥까지
```

### 자주 쓰는 스케줄러들

#### StepLR — 정해진 epoch마다 lr 줄임

```python
from torch.optim.lr_scheduler import StepLR

scheduler = StepLR(optimizer, step_size=10, gamma=0.5)
# 10 epoch마다 lr을 절반으로

for epoch in range(n_epochs):
    train()
    scheduler.step()
```

#### CosineAnnealingLR — 부드럽게 줄임 (현대 표준)

```python
from torch.optim.lr_scheduler import CosineAnnealingLR

scheduler = CosineAnnealingLR(optimizer, T_max=n_epochs)
# 코사인 곡선처럼 lr이 줄어듦
```

#### ReduceLROnPlateau — 검증 점수 안 좋아지면 줄임

```python
from torch.optim.lr_scheduler import ReduceLROnPlateau

scheduler = ReduceLROnPlateau(
    optimizer, mode='min', factor=0.5, patience=5,
)

for epoch in range(n_epochs):
    train()
    val_loss = evaluate()
    scheduler.step(val_loss)    # ← val_loss 전달
```

가장 실용적. **5 epoch 동안 val_loss가 안 좋아지면 lr ÷ 2.**

#### Warmup — 처음에 천천히 시작

Transformer에서 표준. 처음 몇 epoch은 lr을 0에서 시작해서 점진적으로 키움.

```python
# Hugging Face 등의 라이브러리 자주 제공
from transformers import get_linear_schedule_with_warmup

scheduler = get_linear_schedule_with_warmup(
    optimizer,
    num_warmup_steps=1000,
    num_training_steps=n_total_steps,
)
```

## 6. 학습률 찾기 (LR Range Test)

"적절한 lr이 뭔지 모르겠어요" 하는 분들을 위한 트릭.

### 방법

1. lr을 매우 작게 시작 (1e-7)
2. 매 step마다 lr을 점점 키움 (1e-7 → 1.0)
3. 각 lr에서의 loss를 기록
4. 그래프 그려서 "loss가 가장 빠르게 떨어지는 lr"을 찾음

```python
from torch.optim.lr_scheduler import LambdaLR
import matplotlib.pyplot as plt

# lr을 지수적으로 증가시킴
init_lr = 1e-7
final_lr = 1.0
n_steps = 100

scheduler = LambdaLR(
    optimizer,
    lambda step: (final_lr / init_lr) ** (step / n_steps),
)

losses = []
lrs = []

for step in range(n_steps):
    x, y = next(iter(train_loader))
    optimizer.zero_grad()
    loss = loss_fn(model(x), y)
    loss.backward()
    optimizer.step()
    
    lrs.append(optimizer.param_groups[0]['lr'])
    losses.append(loss.item())
    scheduler.step()

# 그래프
plt.plot(lrs, losses)
plt.xscale('log')
plt.xlabel('Learning Rate')
plt.ylabel('Loss')
plt.show()

# loss가 가장 빠르게 떨어지는 lr 찾기 (보통 최저점의 1/10)
```

이 방법은 **fast.ai에서 강력히 추천**하는 트릭이에요.

## 7. 함정과 디버깅

### "Loss가 nan이 됐어요"

99% 학습률 때문이에요.

```python
# 시도 순서
1. lr을 ÷ 10 (0.001 → 0.0001)
2. weight_decay 줄이기
3. gradient clipping 적용
   torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
4. 데이터에 NaN 있는지 확인
```

### "Loss가 한 epoch 후에 멈춰요"

학습률이 너무 작거나, optimizer.zero_grad() 누락.

```python
# 항상 확인
optimizer.zero_grad()      # 1
output = model(x)          # 2
loss = loss_fn(output, y)  # 3
loss.backward()            # 4
optimizer.step()           # 5
```

이 순서를 안 지키거나 1번을 빼먹으면 학습 망함.

### "그래디언트가 너무 큼/작음"

```python
# 그래디언트 norm 확인
for name, param in model.named_parameters():
    if param.grad is not None:
        print(f"{name}: {param.grad.norm():.4f}")
```

너무 크면 (>10) → lr 줄이거나 clipping
너무 작으면 (<1e-7) → lr 키우거나 BatchNorm 추가

## 정리

```python
# 표준 패턴
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=0.001,
    weight_decay=0.01,
)

scheduler = torch.optim.lr_scheduler.ReduceLROnPlateau(
    optimizer, mode='min', factor=0.5, patience=5,
)

for epoch in range(n_epochs):
    # 학습
    for x, y in train_loader:
        optimizer.zero_grad()
        loss = loss_fn(model(x), y)
        loss.backward()
        optimizer.step()
    
    # 검증
    val_loss = evaluate()
    
    # lr 조정
    scheduler.step(val_loss)
```

**한 줄 요약: AdamW + ReduceLROnPlateau가 99%의 경우 잘 동작합니다.**

## 자주 묻는 질문

> **Q. SGD와 Adam, 정말 어느 게 더 좋아요?**
>
> 학계에서 논쟁 중. 일반적으로:
> - **빠른 실험, 작은 모델**: Adam
> - **큰 모델, 충분한 시간**: SGD + Momentum (잘 튜닝하면 더 좋음)
>
> 실무에선 **AdamW가 거의 안전한 선택.**

> **Q. weight_decay가 lr이랑 따로 노는 거예요?**
>
> Adam에서는 살짝 섞여 있고, AdamW에서는 분리되어 있어요. 그래서 AdamW가 weight_decay 효과가 더 정확해요.

> **Q. 옵티마이저를 중간에 바꿀 수 있어요?**
>
> 가능합니다. 학습 후반에 SGD로 바꾸면 일반화가 좋아진다는 연구도 있어요. 다만 복잡하고 효과는 미묘.
